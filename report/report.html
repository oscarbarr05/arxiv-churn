Predicting Researcher Churn on arXiv

A production-grade pipeline that predicts who stops publishing, explains why through feature selection, PCA/SVD and co-authorship structure, and acts on it with personalized paper recommendations.

Final Report · Introduction to Data Science (Units 8–9) · Universidad Politécnica Taiwan–Paraguay · Oscar Barrios · Repository: github.com/oscarbarr05/arxiv-churn

Abstract. From 295 arXiv cs.CL researchers (churn rate 18.6%) the system learns a leakage-free churn model. A consensus of four feature- selection methods reduces 14 engineered features to 8; a RandomForest reaches F1 0.65 / accuracy 88% under 5-fold cross-validation, rising to F1 0.68 once co-authorship centralities are added. An SVD collaborative filter (84% explained variance) recommends re-engagement papers, gated at churn probability ≥ 0.5. The whole system is reproducible with a single command (docker-compose up --build) — no API key, no manual steps. All code, data, the executable notebook and an interactive dashboard are in the GitHub repository.

1 · Problem, data and workflow

Mirroring industrial retention systems (Netflix, Spotify, LinkedIn), the pipeline has three goals — predict churn, explain it, and act on it. The entity is an author publishing in arXiv's cs.CL category; all data is drawn from the public arXiv API (no key, no login). A two-stage scraper first builds an unbiased sampling frame (100 papers per quarter, 2014–2025 ≈ 4 800 papers — sampling across all years prevents a recency-biased cohort with no churners), then fetches each discovered researcher's complete publication history (20 333 author-paper rows), rate-limited at 3 s/request with defensive parsing and pagination.

Figure 1. End-to-end workflow — every stage is implemented as a single-responsibility module in the repository.
Churn label — and why it is leakage-free. A researcher has churned if they submitted no paper after the cutoff 2024-06-30 (≈ 2 years of silence). Features are computed only from pre-cutoff papers and the label only from post-cutoff activity, so recency is a legitimate predictor instead of leaking into the target — exactly how production churn systems frame the task. The 18.6% positive rate sits safely above the 10% danger zone; eligibility requires ≥ 3 pre-cutoff papers and ≥ 1 year of history, and every model also uses class_weight="balanced".

2 · Feature engineering — “better features beat better algorithms”

No raw API field is used directly. Fourteen features span four families, deliberately covering recency, frequency and magnitude (complete definitions and code in the repository):

3 · Feature selection — four methods and a consensus

Four complementary methods are run and consolidated: Filter (variance threshold, correlation pruning at |r| > 0.9, ANOVA F-test), Wrapper (RFE over logistic regression, top 5), and two Embedded rankings (DecisionTree max_depth=5, RandomForest n_estimators=100). A feature survives on ≥ 2 of 4 votes; the same rule ships in model.py, so the notebook analysis and the served model can never diverge.

The disagreements are the insight. papers_per_year ranks near-last under ANOVA (which scores features in isolation) yet is selected by RFE and ranked top-5 by both trees — its signal is interaction-only, precisely what univariate filters miss. is_multidisciplinary is the mirror case (high ANOVA, last in RF): redundant once n_categories is present. When a single tree and the forest disagree, the average of 100 trees is the more reliable referee. The final 8 features are validated by 5-fold stratified CV: accuracy 0.885, precision 0.746, recall 0.582, F1 0.652.

4 · Dimensionality reduction — PCA and SVD

Whereas selection picks existing features, PCA and SVD create compressed ones (PCA on standardized features is equivalent to the SVD of their covariance matrix).

Figure 2 · PCA elbow. Cumulative explained variance against the number of components: two components retain ≈ 58%, three ≈ 70%. The knee motivates keeping only a few components for visualization.
Figure 3 · PCA churn map. The 295 researchers projected to two components, colored by churn. Churned authors (red) separate from active ones (green) along PC1 (the recency axis) — visible separation is direct evidence the features carry signal.
Figure 4 · SVD of the researcher × category matrix (scipy.sparse.linalg.svds, k = 12, 84% variance). Left: scree plot — the leading factor alone captures ≈ 70% of the structure. Right: authors (circles, by churn) and candidate papers (gold diamonds) embedded in the recommender's latent space, where proximity encodes shared research interest and drives the recommendations.
PCA components are weaker model inputs. Training the classifier on three PCA components costs 0.115 F1 (0.537 vs 0.652) and lowers recall to 0.473: PCA compresses for variance, not for churn discrimination. Its value is explanatory (Figures 2–3), not as the production feature set.

5 · Network analysis as feature enrichment

A co-authorship graph is built with networkx (an edge links two researchers who share a pre-cutoff paper). Degree, betweenness and PageRank centralities become new features, and all four selection methods are re-run with them included.

Figure 5 · Co-authorship network. 295 researchers and 417 collaboration edges (node size = degree). Sparse, low-degree nodes on the periphery skew red: isolated researchers churn more — collaboration is, in effect, retention.
Figure 6 · RandomForest importance. Recency (33%) and recent-output share (19%) dominate the served model — the quantitative echo of the four selection methods.
Effect on the model. No individual centrality wins the consensus vote alone — their predictive signal emerges jointly with the temporal features — yet together they raise F1 from 0.652 to 0.678 and recall from 0.582 to 0.618.

The API serves setup A rather than the marginally stronger C: every input of A is computable from a single researcher's public record, whereas C would require recomputing graph centralities on each request, coupling inference to a full-dataset refresh. Setup C is therefore the offline “analyst's model” — a deliberate, documented portability trade-off.

7 · Serving and recommendation

The final RandomForest (n_estimators=300, class_weight="balanced") is persisted with joblib and loaded once at start-up; main.py stays thin, holding only endpoints. The recommender factorizes the researcher × category matrix with truncated SVD and returns the most cosine-similar recent papers by still-active authors — crucially, only for at-risk researchers.

EndpointPurpose
POST /predictChurn prediction → {churned, churn_probability} (8 required features).
POST /recommendTop-N re-engagement papers — only if churn probability ≥ 0.5; otherwise a “not at risk” message.
GET /features · /authors · /healthSelf-documentation, valid author names, and a liveness check.
Example. POST /predict with a dormant profile (recency_days_at_cutoff=600, recent_share_2y=0.1, papers_per_year=0.8…) returns {"churned": true, "churn_probability": 0.58}; sending that author to /recommend yields ranked recent papers, while an active author instead receives the “not at risk” reply — the business rule in action.

8 · Ethics and product reflection

Ethics. A churn score is a probability, not a verdict. Interventions are designed to be harmless under false positives (a relevant-papers digest), and the model must never feed hiring or funding decisions — leaving arXiv is not leaving science. The label is an honest proxy (silence), surfaced through /features, and authors are matched by name, so homonyms remain a disclosed limitation.

Product reflection. As a product manager I would run this as one predict → explain → act loop: score nightly to produce a ranked re-engagement budget; let the feature-selection table route the channel (recency-risk → content nudges, isolation-risk → collaboration features); and treat acting as an A/B experiment that measures incremental retention and feeds the threshold back. The transferable lesson is that churn is rarely sudden — cadence decays first, so monitoring the derivative of engagement allows action while there is still something to save.
