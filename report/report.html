arXiv Researcher Churn Predictor

A production-grade pipeline that predicts which arXiv researchers will stop publishing, explains why through feature selection, PCA/SVD and co-authorship structure, and acts on it by recommending papers to re-engage at-risk authors — the academic analogue of Netflix/Spotify/LinkedIn retention systems.

Final project · Introduction to Data Science · Universidad Politécnica Taiwan–Paraguay (Informatics & Industrial Engineering, Units 8–9) · Author: Oscar Barrios

▶ GitHub repository — github.com/oscarbarr05/arxiv-churn

Public repo. The grader runs a single command: git clone https://github.com/oscarbarr05/arxiv-churn.git && cd arxiv-churn && docker-compose up --build, then opens http://localhost:8000 (dashboard) or /docs (API). No API key, no manual steps — raw data and trained models are committed.

Headline results. 295 researchers (cs.CL), churn rate 18.6%. Leakage-free temporal label. Consensus selection → 8 features. RandomForest 5-fold CV: F1 0.65 / accuracy 88% (original features), rising to F1 0.68 when co-authorship centralities are added. SVD recommender (84% variance) gated at churn probability ≥ 0.5. Fully Dockerized with an interactive dashboard (PCA 2D/3D, SVD space, importance, network).

1. Executive summary — the three jobs

  1. Predict — a RandomForest scores each researcher's probability of going silent, served at POST /predict.
  2. Explain — four feature-selection methods agree that recency dominates; PCA/SVD compress the signal; a co-authorship network shows isolated researchers churn more.
  3. Act — an SVD collaborative filter recommends recent papers by active authors, but only for at-risk researchers (probability ≥ 0.5).

The guiding principle of the unit — “better features beat better algorithms” — is the thread through every section: a plain RandomForest on well-engineered temporal features outperforms anything fancier on raw fields.

3. Data collection — two stages (scraper.py)

Source: the public arXiv API (no key, no login). Collection is split so the churn label reflects real activity, not a sampling artifact:

  1. Sampling frame — 100 papers/quarter, 2014–2025, category cs.CL (~4 800 papers). Sampling across all years avoids the “recent papers only” bias that would exclude churned researchers entirely.
  2. Full histories — for every researcher appearing ≥ 2× in the frame, one au:"Name" query fetches their complete record (20 333 author-paper rows).
Engineering decisions. Rate-limited at 3 s/request (arXiv terms of use); missing fields read defensively with .get(); pagination via start; network errors retried once. Raw output saved to data/raw/ and committed so the build needs no network.

4. Churn label — definition & why it is leakage-free

Definition. A researcher has churned if they submitted no arXiv paper after the cutoff date 2024-06-30 (≈ 2 years of silence at evaluation time).

A naïve days_inactive > 180 label computed “today” would leak recency into both the label and the features, inflating every metric. Instead we split time itself:

  • Features are computed only from papers up to the cutoff (recency is measured at the cutoff, so it is a legal predictor).
  • The label looks only at activity after the cutoff.

This mirrors production churn systems: observe behaviour up to “today”, predict inactivity over the next window. Two years fits academia's rhythm (active cs.CL authors publish with median gaps well under a year, verified in the notebook).

Class balance. 18.6% churned (55 churned / 240 active) — comfortably above the ~10% danger zone. Eligibility filters: ≥ 3 pre-cutoff papers and ≥ 1 year of history. Every model also uses class_weight="balanced" as a second guard.

Run in the notebook and shared with production via model.py::consensus_selection (notebook and API cannot drift apart):

  2. Wrapper — RFE over LogisticRegression, top 5.
  3. Embedded — DecisionTree importances (max_depth=5).
  4. Embedded — RandomForest importances (n_estimators=100).

Reading the disagreements. papers_per_year is ranked 10th by ANOVA (which scores features in isolation) yet selected by RFE and top-5 by both trees — its signal only appears in combination, exactly what filters miss. is_multidisciplinary is the mirror image: ANOVA ranks it 4th but RF last, because given n_categories it is redundant. When the single tree and the forest disagree, we trust the average of 100 trees over one greedy tree.

Final set (8): recency_days_at_cutoff, recent_share_2y, avg_gap_days, n_categories, papers_per_year, career_years, categories_per_paper, solo_ratio. Validated by 5-fold stratified CV — accuracy 0.885, precision 0.746, recall 0.582, F1 0.652.

7. Dimensionality reduction — PCA & SVD

The four methods select existing features; PCA/SVD create new compressed ones. PCA on standardized features ≈ SVD of the covariance matrix.

Figure A. PCA elbow — cumulative explained variance. 2 components ≈ 58%, 3 ≈ 70%.
Figure B. Researchers in PCA space colored by churn. Visible red/green separation along PC1 (recency) = strong features.
Figure C. Truncated SVD of the researcher×category matrix (scipy.sparse.linalg.svds, k=12, 84% variance). Left: scree. Right: authors (circles) and candidate papers (gold) in the recommender's latent space.
PCA as inputs vs originals. Training on 3 PCA components costs 0.115 F1 (0.537 vs 0.652) and drops recall from 0.582 to 0.473: PCA compresses for variance, not churn discrimination. It earns its keep as explanation (Figures A–B), not as the production feature set.

8. Train & serve (model.py, main.py)

Final model: RandomForest (n_estimators=300, class_weight="balanced", random_state=42), saved with joblib and loaded once at startup. main.py is thin — endpoints only. Endpoints:

  • POST /predict — Pydantic input → {"churned": bool, "churn_probability": float}. The 8 model features are required; the other engineered features are optional extras.
  • GET /health → {"status":"ok"}.
  • GET /features — self-documents the expected fields, churn definition and CV metrics.
  • GET /authors, POST /recommend — see §10.

9. Docker — one command from a clean clone

Dockerfile: python:3.11-slim, requirements copied before code (layer caching), uvicorn on port 8000. docker-compose.yml: one churn-api service, ports 8000:8000, ./data mounted.

Robustness decisions. pip runs with --timeout=120 --retries=10 --prefer-binary so the build survives slow networks; the image installs a runtime-only requirements-docker.txt (no matplotlib — only the notebook needs it), cutting ~8.5 MB and 6 wheels. model.pkl, recommender.pkl, viz.json and the raw CSVs are committed, so the container works offline on first run.

10. Recommendation engine (recommender.py, /recommend)

The researcher×category interaction matrix is factorized with truncated SVD (k=12, 84% variance). An at-risk researcher and every candidate paper live in the same latent space; /recommend returns the most cosine-similar recent papers by still-active authors, never the researcher's own work.

Business rule. Recommendations are returned only when churn probability ≥ 0.5; engaged researchers get a “not at risk” message. Re-engagement budget belongs on the at-risk segment.

11. Network analysis as feature enrichment

A co-authorship graph (networkx) is built where an edge means two researchers share a pre-cutoff paper. degree, betweenness and PageRank centralities become new features, and all four selection methods are re-run with them included.

Figure D. Co-authorship network — 295 researchers, 417 edges (node size = degree). Isolated, low-degree nodes skew red.
Figure E. RandomForest feature importance — recency (33%) and recent-output share (19%) dominate.
Honest nuance. No single centrality wins the consensus vote on its own — their signal emerges jointly with the temporal features (the same filter-vs-interaction lesson as papers_per_year). Yet added to the model they lift F1 from 0.652 to 0.678 and recall to 0.618. Sociological reading: collaboration is retention — isolated researchers churn more.

13. Retention interventions tied to the top features

  • High recency / falling recent-share → early-warning “new in your subfield” digest (exactly what /recommend produces) the moment cadence decays, not after two silent years.
  • Low degree / PageRank → collaboration matchmaking (co-authors, workshops) for isolated researchers.
  • has_long_break → soft re-onboarding (“what you missed”) after known gaps.
  • High first_author_ratio + short career → the PhD-attrition profile; institutional mentoring matters more than content nudges.

14. Ethics — acting on predicted, not confirmed, churn

  • False positives must be harmless: a relevant-papers digest is fine; this model must never feed hiring/funding/evaluation. Leaving arXiv ≠ leaving science.
  • Label honesty: the ground truth is arXiv silence, a proxy — stated in the report and returned by /features.
  • Name ambiguity: authors are string-matched; homonyms can merge records — disclosed as a limitation, and a reason not to make public individual claims.

15. PM reflection — if this were Netflix/Spotify

Treat the pipeline as one predict → explain → act loop. (1) Predict runs nightly, producing a ranked re-engagement budget, not a list of “doomed users”. (2) Explain routes the channel: recency-risk → content nudges (/recommend); isolation-risk (network features) → social features. The feature-selection table is literally the intervention router. (3) Act is an experiment, never a fiat: hold out a control group, measure incremental retention, feed it back into the threshold. The 0.5 gate is the first, deliberately simple version of that policy. The deepest lesson transfers across domains: churn is rarely sudden — cadence decays first, so systems that watch the derivative of engagement act while there is still something to save.

17. Reproducibility, limitations & repository

Rebuild from scratch: scripts/collect_data.py → build_model.py → scripts/make_viz.py → scripts/make_notebook.py. Stack: Python 3.11, FastAPI, scikit-learn, pandas, numpy, scipy, networkx, matplotlib, joblib; dashboard uses vendored Plotly (offline).

Limitations: arXiv silence is a proxy for leaving research (venue migration looks like churn); author names are not disambiguated (no ORCID); one category (cs.CL); graph centralities are community-relative, not global.
